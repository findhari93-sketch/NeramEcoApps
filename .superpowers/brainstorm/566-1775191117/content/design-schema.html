Key Relationships:
course_plans → classroom (1:many plans per classroom)
plan_weeks → course_plan (belongs to one plan)
plan_sessions → plan_week + topic + teacher + scheduled_class (the bridge to timetable)
plan_homework → session (homework belongs to a day's session)
homework_submissions → homework + student (one per student per homework)
plan_tests → plan + week + QB test (the bridge to question bank)
plan_resources → topic OR session (dual attachment point)

Key Design Decisions

Why this structure?

Sessions link to scheduled_classes — When a teacher "pushes" a session to the timetable, we create a nexus_scheduled_class and save its ID. This is the bridge between plan and timetable.

Tests link to QB test system — Weekly tests connect to your existing question bank. Teachers select questions from recalled papers and link them.

Homework is per-session, not per-day — Since AM and PM are different teachers/topics, homework is tied to the specific session. "After Colour Theory (AM), draw 2 colour wheels."

Resources have dual attachment — Topic-level (always visible) vs session-level (today's resources). topic_id XOR session_id.

Reuses existing tables

✅ nexus_classrooms — plan belongs to a classroom

✅ nexus_topics — sessions reference existing topics

✅ nexus_scheduled_classes — pushed sessions become real classes

✅ nexus_attendance — attendance tracked on pushed classes

✅ users — teachers and students

✅ QB test system — weekly tests link to existing test engine

No duplication — course plan is an orchestration layer on top of your existing system.
